我 2026 年到目前为止的经历非常奇妙。

年初，因为被老板雪藏，每天被塞各种垃圾活儿，我一度认真考虑过离职退休。那段时间，我基本处于摆烂状态，每天就等着公司哪天大发慈悲，给我发一个“毕业大礼包”。

事情的转折，出现在公司给我开通 GitHub Copilot 账户之后。

有了 AI Agent，老板安排的那些垃圾活儿很快就能干完。剩下的时间，我终于可以拿来做自己真正感兴趣的事情。那种感觉，仿佛一切又回到了读博士的时候：可以自由探索，可以折腾工具，可以把精力花在真正有意思的问题上。

于是，摆烂的心一下子就没了。现在每天恨不得早点去公司烧 token。

token 真是打工人的毒品，一旦沾上，就再也离不开了。

## 何时能实现AI for Hardware Design

![image-1](/assets/when-eda.png)

众所周知，当 AI 在软件编程领域大杀四方，甚至已经隐隐有让万千软件工程师失业的势头时，它在硬件设计领域，比如 IC、FPGA、PCB 设计方面，依然还是个弟弟。

这也没有办法。LLM 在硬件设计领域的 training data 实在太少了。

我认为，这种现状会一直持续到我在之前文章中介绍过的 EDA CI/CD 系统被大规模推广之后。在没有 EDA CI/CD 的情况下，工程师的设计过程大多仍然停留在本地环境里。于是，下面这些对训练 AI 最重要的数据，就很难被系统性地蒸馏出来：

* 为什么选择某种架构？
* 工程师在本地做了哪些改动？
* 为什么要做这些改动？
* 他们期待每次改动带来什么效果？
* 仿真结果是否符合预期？
* 如果不符合预期，下一步应该怎么修改？

只有当 EDA CI/CD 被大规模推广之后，工程师的每一次设计迭代，以及相关上下文数据，才有可能被 push 到 cloud 上，并最终沉淀为 AI 的训练数据集。

有了海量、高质量、带上下文的设计数据，AI 才有可能真正学会设计电路。

所以，从一个硬件设计者的角度来看，在 AI 具备独立硬件设计能力之前，真正的王道并不是幻想它立刻取代我们，而是先学会使用 AI 优化工作流程，让 AI 帮我们干那些脏活、累活、重复活。

在接下来一段时间里，我会通过一系列文章，展示我是如何在日常工作中践行这一理念的。

今天，我将先用本周完成的两个例子，来说明我是如何使用 GitHub Copilot 干活的。

## 解析 FPGA Flash Dump

![image-2](/assets/fpga-dump-debug.png)

我现在负责的产品里有一块 Xilinx Artix-7 FPGA。它的 programming file 存在一片 32 MB 的 flash 上。为了支持 multiboot，整个 programming file 被分成了 4 个分区：

- Golden：包含系统启动以及识别外接硬件类型所需的基本配置。
- Config A：包含适配外接硬件 A 的配置文件。
- Config B：包含适配外接硬件 B 的配置文件。
- Config C：包含适配外接硬件 C 的配置文件。

这款产品之前在中国国内生产时良率一直很高。自从公司响应川建国的 MAGA，把工厂搬回美国以后，就开始出各种幺蛾子，良率始终上不去。

去工业化这件事，真的是一旦刀枪入库、马放南山，再想把传统手艺活儿捡起来，就没那么容易了。

这几天，负责测试的工程师 S 报告说，他们收到美国代工厂发来的产品后，更新完固件，也就是 FPGA 和 MCU 之后，发现系统无法识别外接硬件 A/B/C。

根据经验，我的第一反应是：FPGA 的 programming file 大概率坏掉了。

于是，我用AI写了一个 Tcl 脚本发给 S，让他把每天坏掉产品里的 FPGA programming flash 内容 dump 出来，再发给我分析。

做 FPGA 就是这样。大部分时间其实都在写文档、debug、维护现有产品，干这些脏活累活。真正从零开始设计产品，只占日常工作中很小的一部分。

要是搁以前，我拿到这些 dump 文件之后，会打开 Beyond Compare，一处一处地比对正确文件和 S 传回来的 dump，直到眼睛干涩、布满血丝。

这种方法不但耗时间，而且非常容易遗漏文件损坏的 pattern。

现在有了 AI，我要是还这么干，那我就是傻子。

收到 25 个 dump 文件以后，我祭出了 GitHub Copilot，然后选取了 Claude Opus 4.7，给它定义了下面这组任务：

- 建立分析用的文件夹结构。
- 从 JFrog Artifactory 上获取这个工程 release/xxx 对应的 FPGA implementation artifact，并从中提取 bin 文件，作为后续比对的参考文件。
- 从 SharePoint 上下载 S 上传的 flash dump 文件。
- 解析每个 dump 文件，读取对应的硬件编号，并将硬件编号和 dump 文件说明里的故障描述提取出来，整理到一个 Excel 表格中。
- 将每个 dump 文件和参考 bin 文件进行比对，记录错误 pattern、出现位置和出现频率。
- 完成比对之后，自动调用公司的 PowerPoint 模板，生成一份分析报告。
- 打开 Outlook，写一封邮件，把分析报告发给 A/B/C 等相关人员。
- 整理它在整个过程中生成的 Python 脚本和 PowerShell 脚本，并给我写一份教程，方便我以后学习对应内容。

交代完这些之后，GitHub Copilot 就开始安静地干活了。而我则可以安心继续搞自己感兴趣的事情，中间只需要时不时过来批准一下它的请求。

AI 瞬间让我有了一种被升职到老板那个位置的错觉：接到任务，拆分好，丢给下属，然后时不时看看进度，批准一下他们的 request 就行。

![image-1](https://static04.hket.com/res/v3/image/content/465000/467004/known_1200.jpg)

我不能从公司电脑上截图，所以只能在这里口述一下 Copilot 替我干活时都做了什么：

1. 写了一个 Python 脚本，从 SharePoint 上把 25 个 dump 文件逐个下载到本地。
2. 写了第二个 Python 脚本，将每个 dump 文件和参考 bin 文件做逐 bit XOR，把非零 bit 的位置记录下来，并存成 diff-dump-xx.bin 文件。
3. 写了第三个脚本，分析 diff-dump-01.bin 到 diff-dump-25.bin，统计每个分区里 bit error 出现的 pattern 和概率，并写入 Excel 表格。
4. 写了第四个脚本，把前三步得到的结论汇总成一份 12 页的 PowerPoint 报告。
5. 调用 PowerShell 自动生成邮件。最后我只需要检查一下，然后点击发送。

从结果来看，Copilot 整体的分析质量已经达到了我手工分析的水平。

它准确地发现，所有 flash dump 文件里的 Golden 数据都完好无损。因此可以得出结论：FPGA 在不连接外接硬件 A/B/C 的情况下，应该是可以正常工作的。

它也准确地发现，flash dump 文件里存放 Config A/B 数据的区域存在大量随机错误。从 byte 级别来看，1-bit error 占多数。也就是说，这些出错 byte 和参考 bin 文件中对应的 byte 通常只有 1 个 bit 发生了翻转。

它还准确地发现，flash dump 文件里存放 Config C 数据的区域基本全部损坏。看起来这一部分几乎完全没有成功写入。

除此之外，它还列出了导致上述 flash 写入错误的几种可能原因，并给出了后续 debug 的思路。

在我看来，这已经足够我在每天的 debug 进度会上讲 15 分钟了。

唯一的不足是，Copilot 在 PPT 里贴的分析图尺寸太大，把文字都遮住了。这个问题当然也能继续让它改，但即便如此，整体效果已经非常惊人。

如果是我自己手动分析，我觉得三天都不一定能把这些活儿干完。毕竟我还得处理其他事情。而且就算三天以后干完，结果也不一定比它更让我满意。

AI 一个小时能干成这样，此时此刻，我只能祭出下面这张梗图：

## 制作 FPGA 培训教程

![image-3](/assets/ai-coaching.png)

在公司里，带新人是一件最没有 visibility 的事情。

带好了，那是你应该的；带不好，出了问题还要拿你是问。所以在公司里，大家一般都不太喜欢带新人。

这不，我们部门来了一个学医学技术的本科生实习生。大家都不愿意带，于是这活儿又被塞给了我。

这孩子在学校学的都是高级医疗技术，没怎么碰过 FPGA，所以我还得从头开始手把手教。好不容易教会他用 VHDL 写一些基本逻辑之后，我就跟他说：

你上网看看，怎么把这段代码打包成一个 AXI Lite IP 吧。

结果他上网找了半天，最后跟我说，没有找到适合我们开发板的教程。然后他把找到的链接发给我。

我一看，好家伙，这教程是 2014 年写的。那个时候，这孩子可能还在读小学。而且这个教程用的是 Zynq 开发板，而我们手头只有一块 AC701。

我实在懒得自己手动爆改这个教程，于是直接把链接丢给 GitHub Copilot，然后让它把这份教程改成适配 AC701 和 Vivado 2020.2 的版本。

GitHub Copilot 两分钟就把教程搞定了。最后，它在本地文件夹里生成了以下内容：

一份 Markdown 格式的教程，包含详细步骤，以及每一步背后的逻辑说明。
一个适用于 AC701 的 Tcl 文件。使用者只需要运行这个 Tcl 文件，就能自动把工程建好。
生成的工程没有使用 Zynq，而是改成了用 AC701 上的 MicroBlaze IP 来搭建测试系统。
最后还把整个文件夹打包成 zip，方便直接发给实习生。

整个过程，我没有做任何人工干预。

实习生那边拿到教程以后，很快就可以开始上手。不过因为他对 Vivado 实在不熟，很多按钮都找不到位置，于是又时不时在 Teams 上问我：这个按钮在哪里，那个菜单怎么打开。

我这边正忙得焦头烂额，于是就跟他说：

你把这个教程导入 Copilot，后面遇到哪里不会，直接问 Copilot 就行了。

从那以后，他再也没有来问我。

过了半个小时，我去厨房喝水，路过他的工位，就顺便看了一下他的进度。结果他说早就做完了。

他对 Copilot 的评价是：问啥答啥，恨不得把每一步都揉碎了、掰开了讲，态度好得不得了。

此外，我看了一下他和 Copilot 的对话，让我觉得比较好的一点是：Copilot 的回答非常简短切题，不会像 ChatGPT 那样，问一个简单问题，结果甩出来一大段百科全书式回答。

我觉得，AI 普及以后，做教培的人的日子恐怕就不好过了。

Copilot 提供的这种定制化培训、1v1 辅导和实时答疑，是任何规模化培训班都很难比得上的。
